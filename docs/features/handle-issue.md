# 기능 명세서: handle-issue 워크플로우

## 1. 개요 및 목적
`handle-issue` 워크플로우는 특정 GitHub 이슈에 대한 분석, 설계, 구현, 리뷰 및 PR 생성까지의 모든 개발 생명주기를 자동으로 처리하는 종합 오케스트레이션 파이프라인입니다.

## 2. 사용법 (Trigger)
- **명령어**: `/handle-issue #<issue-id>`
- 보통 스케줄러(cron) 등에 의해 백그라운드 스크립트(`scripts/handle-issue.sh`)로 자동 실행되거나 CLI를 통해 명시적으로 호출됩니다.

## 3. 주요 동작 방식 및 내부 로직
0. **워크플로우 메타 플래닝**: 작업 시작 전, 전체 단계(1~12)에 대한 체크리스트를 작성하여 필수 단계(격리, 설계, 반복 리뷰 등)가 누락되지 않도록 자기암시(Self-Suggestion)를 수행합니다.
1. **이슈 검증**: 대상 이슈가 `status: confirmed` 상태인지, 그리고 `status: in-progress`가 아닌지 확인합니다.
2. **작업 환경 격리**: `git worktree add .worktrees/<branch-name>`을 통해 메인 환경과 분리된 독립적인 작업 트리를 생성합니다.
3. **분석 및 설계**: 필요 시 `@research`를 호출하고, 필수적으로 `@design` 스킬을 호출하여 설계안(`design_report.md`)을 산출합니다.
4. **구현 및 리뷰**: 설계안에 따라 코드를 수정하고, 작성된 코드를 `@review` 스킬을 통해 검증합니다. 피드백을 반영하여 코드를 반복적으로 수정합니다.
5. **PR 생성**: 코드의 품질 기준이 충족되면 `@pr-create` 스킬을 호출하여 변경사항을 푸시하고 PR을 생성한 후 작업 환경을 정리합니다.

## 4. 제약 사항 및 예외 처리
- **환경 격리 강제**: 모든 변경사항은 반드시 `.worktrees/` 내에서 수행되어야 하며 메인 소스 코드를 직접 변경해서는 안 됩니다.
- 각 스킬이 생성하는 보고서는 지정된 `tmp/` 폴더 등에 아티팩트로 저장되어야 합니다.
