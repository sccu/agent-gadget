---
description: Automatically handles a GitHub issue from branch creation to implementation, AI review, and PR creation using the pr-create skill.
---

> [!IMPORTANT]
> **Isolation Rule**: This workflow is designed to execute in a dedicated Git worktree (`.worktrees/`) to prevent interference with the main workspace. This isolation is mandatory for automated issue handling but does not apply to direct prompt-based tasks on the `main` branch.


1. Ask the user: "Please provide the GitHub Issue ID or Issue Number you would like me to work on." Wait for their input.
2. Invoke the `@gh-issue` skill to verify the issue and update its status. If the skill reports a verification failure, halt the workflow and inform the user.
3. Determine an appropriate branch name containing the issue identifier (e.g., `issue-<id>-brief-description`).
4. Run command to isolate your work in a new worktree: `git fetch origin main && git worktree add .worktrees/<branch-name> -b <branch-name> origin/main && cd .worktrees/<branch-name>`. **CRITICAL**: All subsequent steps MUST be performed within this worktree directory to ensure environment isolation. Always use relative paths when referencing files within the project to maintain portability.
5. If necessary, invoke the `@research` skill with the issue number. It is highly recommended to invoke this skill if previous attempts have failed, or if there is insufficient information in the codebase and documentation to fulfill the request.
6. Invoke the `@design` skill with the issue number (**MANDATORY**: do not skip).
7. Based on the `research_report.md` (if available) and `design_report.md` artifacts, and the issue content, write a detailed implementation plan. **Note**: All generated reports and by-products MUST be saved in the system-designated artifact directory (e.g., `tmp/` directory provided in metadata), never in the project root or source directories.
8. Implement the code changes exactly according to the plan created in the previous step.
9. Once your implementation is ready, invoke the `@review` skill with the issue number to analyze your changes.
10. Read the review feedback artifact (e.g., `review_report.md`) generated by the reviewing agent and iteratively refactor the code until it meets the required quality standards.
11. If the review artifact contains **no issues and no items requiring user review**, invoke the `@pr-create` skill to automatically push the branch, create a pull request, and clean up the workspace. Provide the issue ID as a parameter to the skill.
12. Finally, present the results to the user, including the Issue URL and Pull Request URL outputted by the `@pr-create` skill.
